## 應用場景說明

本指南的核心應用場景是：**當你身處中國內地時，希望通過位於境外地點的家庭網絡，安全、穩定地訪問境外互聯網資源。**

一個典型的具體場景是：**外國友人來中國旅行時，希望避免昂貴的國際數據漫遊費用，同時維持與境外必要的通信（如電子郵件、即時通訊、社交媒體等）**。通過在境外的住所部署一台配置了本指南所述服務的路由器，旅行者可以在中國內地通過該路由器安全地中繼網絡流量，既保持了通信的暢通，也避免了直接使用可能存在風險的公共 Wi-Fi 或付費 VPN 服務。

本指南以境外（如新加坡、香港）節點為例進行配置，但服務器端的部署原理適用於任何具備公網 IP 的境外網絡環境。

## 1. 網絡拓撲與前置準備

### 1.1 硬體與網絡環境

| 組件 | 規格要求 |
| :--- | :--- |
| **服務端（對端節點）** | 位於境外的住宅寬頻環境，具備公網 IPv4，且強烈建議開通原生 IPv6 雙棧。 |
| **網關端（本地）** | 支持 OpenWrt 系統的路由器（本文以 GL-MT6000 為例）。 |
| **客戶端設備** | Windows PC（`v2rayN`）、Android 移動設備（`v2rayNG`）。 |

### 1.2 關鍵物料準備：實體網線 (RJ45)

在對本地路由器進行 OpenWrt 固件刷寫時，**必須準備一條實體網線（RJ45，即常見的水晶頭網線）將 PC 與路由器的 LAN（區域網絡）接口直接相連**。

> **工程提示**：多數第三方 OpenWrt 固件在初次刷入並完成初始化重啟後，其 `radio0/radio1`（Wi-Fi 模組）預設為關閉狀態（Disable），或 SSID（服務集標識符）配置被覆蓋。若完全依賴無線網絡進行刷機，在重啟階段將會丟失物理層連接，導致無法進入後台管理介面。


## 2. 網關層：OpenWrt 刷寫與初始化部署

### 2.1 固件刷寫流程

1. 將 PC 端 IP 地址設置為靜態 IP（網際協議地址），通常為 `192.168.1.x` 網段，具體依賴於路由器的 U-Boot（引導加載程序）/底層網段設定。
2. 使用**實體網線**連接 PC 與路由器的 LAN 口，確保物理鏈路（Link State）處於 UP 狀態。
3. 進入路由器的原廠後台或 U-Boot/Breed 恢復模式，上傳 OpenWrt `.bin` 或 `.img` 固件檔案。
4. 確認刷寫並等待設備自動重啟（期間切勿斷電）。

**固件下載與版本選擇**：

- **官方下載地址**：
  - OpenWrt 官方固件選擇器：https://firmware-selector.openwrt.org/ （搜索 GL-MT6000）
  - GL.iNet 官方適配版：https://dl.gl-inet.cn/router/mt6000/open

- **版本區別與建議**：
  - **主線穩定版 (23.05.x)**：穩定性極高，適合長期運行，但無線驅動可能較舊。
  - **新版開發版 (Snapshot / 24.10)**：包含最新無線驅動，性能和兼容性更好，但可能不如 23.05.x 穩定。
  - **建議**：追求極致穩定選 **23.05.x**；追求新功能與性能選 **24.10.x** 或 GL.iNet 的 **op24** 版本。

> **文檔時間戳**：本文檔更新於 **2026年6月**，建議讀者訪問官網獲取最新版本信息。

### 2.2 網絡初始化與分流配置

1. 固件啟動後，通過瀏覽器訪問預設網關（通常為 `192.168.1.1`）。
2. 進入 **網絡 (Network) -> 無線 (Wireless)**，手動啟用 Wi-Fi 模組並配置 SSID 與加密協議，此時方可斷開實體網線。
3. **（可選）安裝中文語言包**：若希望 PassWall2 界面顯示中文，可在安裝 PassWall2 時一併安裝語言包：
   ```bash
   opkg install luci-i18n-passwall2-zh-cn
   ```
   安裝完成後刷新 LuCI 頁面，界面即切換為中文。若使用一鍵安裝腳本，部分腳本會自動安裝語言包，可無需手動操作。
4. 在 OpenWrt 中安裝並配置 Xray 客戶端插件，推薦使用 **PassWall2** 或 **OpenClash**。
   - 若 `opkg`（OpenWrt 包管理器）無法找到 `luci-app-passwall2` 包，可執行一鍵安裝腳本：
     ```bash
     cd /tmp && rm -f passwall2.sh && wget -O passwall2.sh https://raw.githubusercontent.com/enxy0/passwall2_install/main/passwall2.sh && sh passwall2.sh
     ```

> **⚠️ 重要提醒**：每次在 OpenWrt LuCI 界面中修改配置後，**必須點擊頁面右下角的 `Save & Apply`（保存 & 應用）按鈕**，否則配置不會生效。

## 3. 傳輸層：VLESS-XTLS-Reality 核心配置

### 3.1 為什麼選擇 VLESS + XTLS + Reality？

在部署跨境網絡傳輸方案時，協議的選擇直接決定了方案的**抗干擾能力**、**傳輸效率**和**部署複雜度**。本指南選擇 VLESS + XTLS + Reality 組合，基於以下考量：

**① 協議層面：VLESS vs VMess**

VLESS（V2Ray Lightweight Encryptionless Stream Protocol）是 V2Ray / Xray 生態中相較於 VMess 更輕量的協議。兩者的核心區別在於：

- **VMess**：協議自身攜帶加密層，客戶端與服務端之間需要進行額外的加密和解密操作。這雖然提供了「開箱即用」的安全性，但也帶來了額外的 CPU 開銷和傳輸延遲。
- **VLESS**：協議本身**不提供加密**，而是將加密工作「外包」給外層的 TLS 傳輸層。這使得 VLESS 協議頭部極其精簡，沒有可被檢測的固定加密特徵，同時減少了服務端的計算負擔。

在路由器（尤其是嵌入式設備）上部署時，VLESS 的輕量化特性對性能更友好。

**② 傳輸層：XTLS (Xray Transport Layer Security)**

XTLS 是 Xray 核心對 TLS 傳輸層的增強實現。其核心優化在於：在代理轉發流量時，**消除冗餘的加密操作**。傳統代理方案中，流量需要經過「解密 → 再加密」的過程；而 XTLS 允許代理直接轉發已加密的流量流，從而降低延遲、提升吞吐量。

**③ 抗檢測機制：REALITY**

REALITY 是 Xray-core 中最先進的抗審查傳輸層方案。其工作原理與傳統 TLS 有本質不同：

- **傳統 TLS**：服務器需要持有有效的 TLS 證書（無論是付費證書還是 Let's Encrypt 免費證書），客戶端驗證該證書後建立連接。防火牆可以通過主動探測（Active Probing）來檢查服務器是否真正持有對應證書的私鑰，從而識別代理服務器。
- **REALITY**：服務器**不需要持有任何 TLS 證書**。它「竊用」一個真實熱門網站（如 `images.apple.com`）的 TLS 指紋。當防火牆主動探測時，服務器會將探測流量轉發至該真實網站，返回真實的 TLS 握手響應。防火牆無法區分這是一個真實網站還是一個代理服務器——因為從指紋到證書鏈，一切都是真實的。

這一設計使 REALITY **免疫於主動探測**，是目前公認抗干擾能力最強的方案之一。

**④ 組合優勢**

VLESS（輕量無特徵）+ XTLS（高效轉發）+ REALITY（免疫探測）三者結合，構建了一套在**性能**和**抗干擾**兩個維度上都達到目前技術前沿的跨境傳輸方案。尤其在深度包檢測（DPI, Deep Packet Inspection）嚴格的網絡環境中，此配置能將代理流量偽裝為標準的 HTTPS 流量，使其極難被檢測和封鎖。

### 3.2 境外服務端配置示例 (`config.json`)

```json
{
  "inbounds": [
    {
      "port": 8443,
      "protocol": "vless",
      "settings": {
        "clients": [
          {
            "id": "<YOUR_UUID>",
            "flow": "xtls-rprx-vision"
          }
        ],
        "decryption": "none"
      },
      "streamSettings": {
        "network": "tcp",
        "security": "reality",
        "realitySettings": {
          "show": false,
          "dest": "images.apple.com:443",
          "xver": 0,
          "serverNames": ["images.apple.com"],
          "privateKey": "<YOUR_PRIVATE_KEY>",
          "shortIds": ["0123456789abcdef"]
        }
      }
    }
  ]
}
```

**配置參數說明**：

| 參數 | 說明 |
| :--- | :--- |
| `port` | 服務器監聽端口。建議使用 1024-65535 之間的冷門端口（如 8443），避免使用 443、8080 等常見端口以減少被針對性掃描的概率。 |
| `protocol` | 必須設置為 `vless`。 |
| `clients[].id` | 客戶端 ID (UUID，通用唯一識別碼)。**在 PassWall2 界面中此字段顯示為「ID」**，其值即為 UUID。服務器與客戶端的此值**必須完全一致**。 |
| `clients[].flow` | 流控模式。`xtls-rprx-vision` 是目前 REALITY 配置下推薦的流控值，啟用 XTLS 的流量直通優化。 |
| `decryption` | VLESS 協議自身不加密，此處必須設置為 `none`。 |
| `network` | 傳輸層協議。`tcp` 是最通用、兼容性最好的選擇。 |
| `security` | 傳輸層安全方案。設置為 `reality` 以啟用 REALITY 抗檢測機制。 |
| `realitySettings.dest` | 偽裝目標。REALITY 會將探測流量轉發至此地址。應選擇與服務器地理位置相近、且訪問量大的知名網站域名（如 `images.apple.com:443`）。 |
| `realitySettings.serverNames` | 服務器名稱列表，通常與 `dest` 的域名保持一致。 |
| `realitySettings.privateKey` | 服務器私鑰，由 `xray x25519` 命令生成（見下方「密鑰生成說明」）。對應的公鑰需配置在客戶端。 |
| `realitySettings.shortIds` | **格式限制**：十六進制字符串（僅含 `0-9`、`a-f` 或 `A-F`），長度為偶數，上限 16 字符。可留空 `""`。可用於區分不同客戶端。 |

> **`shortIds` 深度解釋**：此字段並非用於加密，而是用於在同一個端口後區分多個不同的客戶端，服務端可根據不同的 `shortId` 匹配不同的 `client` 配置。**若你只有一個客戶端，完全可以直接留空 `""`**。錯誤配置會導致連接失敗。

**🔑 密鑰生成說明（重要）**：

在 OpenWrt 的 SSH 終端中執行以下命令生成 REALITY 所需的 X25519 密鑰對：

```bash
xray x25519
```

執行後會輸出類似：
```
Private key: sK9VV_qkrI5Aex7YCsFt9lgddBVK9DIUyzN9R9PYIF9
Public key:  C2jbG73fEkDX190hplzPkDmUZ0vJYowxzZoLu-ylgD9
```

**⚠️ 注意事項**：
- 每次執行 `xray x25519` 生成的 Private Key 和 Public Key 是**唯一配對的**。服務器的 `privateKey` 必須與客戶端填入的 `publicKey` 來自**同一次生成**。
- **Private Key（私鑰）** 填入服務器配置的 `privateKey` 字段，**嚴禁洩露**。
- **Public Key（公鑰）** 填入客戶端（v2rayN / v2rayNG）的 `publicKey` 字段。
- 可多次執行 `xray x25519` 生成新的密鑰對，但每次更換後服務器和客戶端必須同步更新。
- **複製 Public Key 時請注意前後不要含有多餘空格或換行符**，否則客戶端將無法識別。

**🆔 shortIds 生成方式**：

可使用以下命令生成 16 位十六進制 shortId：
```bash
openssl rand -hex 8
```

> **安全建議**：
> - `dest` 與 `serverNames` 參數應選擇與對端 IP 地理位置相近、且未被阻斷的知名網站域名，切勿直接使用容易被關聯的私人域名。
> - 所有 `<YOUR_*>` 佔位符請替換為實際生成的密鑰與 ID (UUID)。

### 3.3 服務端防火牆 (Firewall) 配置

服務端完成設定後，必須在 OpenWrt 防火牆中放行對應端口。

| 操作項 | 說明 |
| :--- | :--- |
| **IPv4 放行** | 進入 `網絡 (Network)` → `防火牆 (Firewall)` → `通信規則 (Traffic Rules)`，新增規則：來源區域 `wan`（廣域網），協議 `TCP`，目標端口 `8443`，動作 `接受 (Accept)`。 |
| **IPv6 放行** | 若使用 IPv6，需在 IPv6 防火牆中新增類似規則。注意 IPv6 不依賴 NAT（網絡地址轉換），需直接放行該端口的入站流量。 |
| **驗證方式** | 在內地環境使用 `telnet [域名或IP] 8443` 測試端口是否可達。 |

> **⚠️ 重要提醒**：每次在 OpenWrt LuCI 界面中修改配置後，**必須點擊頁面右下角的 `Save & Apply`（保存 & 應用）按鈕**，否則配置不會生效。僅點擊 `Save`（保存）不會應用變更。

### 3.4 本節故障排查

> **說明**：本節僅涵蓋服務器端配置階段的排查。客戶端連接相關的排查請參見第 6.3 節。

| 問題現象 | 可能原因 | 解決方法 |
| :--- | :--- | :--- |
| Xray 服務無法啟動 | `config.json` 格式錯誤（如多餘逗號） | 使用 `jsonlint` 或 [JSONLint](https://jsonlint.com/) 校驗配置文件格式。 |
| 服務器端口無法訪問（`telnet` 返回 `Connection refused`） | Xray 服務未運行 / 端口未監聽 | 執行 `netstat -tulpn | grep 8443` 確認服務監聽；若無輸出，執行 `systemctl restart xray` 或 `/etc/init.d/xray restart`。 |
| 端口可達（`telnet` 成功）但客戶端仍失敗 | 防火牆入站規則不完整 / NAT 配置錯誤 | 確認防火牆規則中目標區域為 `Input`（而非 `Lan`），確認外部端口與內部端口一致。 |
| 客戶端測試延遲返回 `failed` | 服務器端口未放行 / 防火牆阻斷 | 在境外服務端執行 `netstat -tulpn | grep 8443` 確認服務監聽；在內地執行 `telnet [域名] 8443` 確認端口可達。 |


## 4. 協議棧與鏈路層優化（核心調優）

在完成基礎連通性測試後，若在晚高峰或特定網路環境下依然存在明顯卡頓或高延遲（如網頁響應耗時超過 60 秒），必須進行以下底層參數的深度優化。

### 4.1 最大傳輸單元 (MTU) 匹配與防分片

- **問題成因**：在虛擬網卡（Tun 模式）下，若上層 MTU 設定過大，數據包在經過 Xray 協議加密封裝後，總長度會超出實體網卡或公網路由器的標準 MTU（1500）。這將導致 IP 數據報文在公網傳輸時被強制分片（Fragmentation）。在晚高峰高丟包環境下，任何一個分片的丟失都會導致整個上層 TCP（傳輸控制協議）報文重傳，引發雪崩效應。

- **參數優化公式**：
  ```
  MTU_tun ≤ MTU_physical - Encapsulation_Overhead
  ```

- **實踐指導**：
  - 若物理網卡 MTU = 1500（普通寬頻），則 Tun MTU 推薦設為 **1350~1400**。
  - 若運營商使用 PPPoE（基於以太網的點對點協議）接入（MTU 限制為 1492），建議進一步下調至 **1300**。
  - IPv6 環境最低可設置為 **1280**（IPv6 強制最小 MTU）。

### 4.2 路由死循環與 DNS 解析超時規避

當客戶端（如 `v2rayN`）開啟 Tun 模式後，經常出現首次加載網頁需等待 1 分鐘以上才響應的現象，通常由以下配置失誤引起：

1. **禁用「嚴格路由 (Strict Route)」** ：必須將其關閉，依賴標準的路由表進行轉發。
   
   > **技術提示**：此選項在部分客戶端中被稱為「路由衝突檢測」或「防止 DNS 洩漏」，其本質都是強制流量走虛擬網卡。在跨境代理場景下，必須關閉它，讓系統自己判斷路由，否則所有本地 DNS 請求都可能被誤導至代理，引發超時。

2. **淨化 DNS 解析鏈路**：確保直連 DNS（如 `223.5.5.5`，阿里雲 DNS）與遠端代理解析 DNS（如 `1.1.1.1`，Cloudflare DNS）分離，並關閉瀏覽器自帶的「安全 DNS (DoH)」，防止瀏覽器繞過系統分流策略。

### 4.3 關閉多路復用 (Mux)

在低丟包的專線環境下，Mux 可減少 TCP 握手次數；但在跨國公網直連環境中，**強烈建議關閉 Mux**。在高丟包率發生時，Mux 內單個 TCP 擁塞控制窗口的收縮會牽連同一通道內的所有並發連接，導致整體吞吐量斷崖式下跌。

### 4.4 本節故障排查

| 問題現象 | 可能原因 | 解決方法 |
| :--- | :--- | :--- |
| 啟用 Tun 模式後首次打開網頁需等待 60 秒以上 | 嚴格路由 (Strict Route) 攔截了 DNS 請求 | 在客戶端設置中關閉「嚴格路由」，改為依賴系統路由表轉發。 |
| 啟用 Tun 模式後國內網站也走代理，速度變慢 | 分流規則未正確生效 | 將路由規則設為「繞過大陸 (Whitelist)」，並確保 GeoIP（地理位置 IP 數據庫）文件已更新。 |
| 下載速度大幅低於帶寬上限 | MTU 過大導致分片重傳 | 按 4.1 節公式下調 Tun MTU 至 1300~1400，逐次測試直至速度穩定。 |


## 5. 網絡層策略：IPv6 雙棧優先級轉發

由於防火牆在 IPv4 邊界部署了高密度的 DPI 設備，晚高峰期間 IPv4 跨境路由往往面臨極端的 QoS（服務質量）降級。

- **原理機制**：利用三大運營商在 4G/5G 蜂窩網絡及部分家庭寬頻中全面普及的原生 IPv6，將目標節點地址由 IPv4 切換為對端的公網 IPv6 地址。這不僅能徹底繞過 IPv4 的 NAT 轉換層，還能大幅降低經過 DPI 設備時的排隊處理延遲。

- **移動端操作（v2rayNG）** ：在 Android 設備連接蜂窩數據或作為熱點發出時，直接將節點配置中的 `Address` 欄位修改為境外服務端的 IPv6 地址（格式如 `[240e:xxxx:xxxx::1]`）。

- **防火牆放行配置**：注意，IPv6 不依賴 NAT 端口映射。必須登入境外端路由器後台，在 IPv6 Firewall 模組中新增 Inbound（入站）規則，直接放行針對該伺服器 IPv6 地址及 `8443` 端口的 TCP 流量。

### 5.1 本節故障排查

| 問題現象 | 可能原因 | 解決方法 |
| :--- | :--- | :--- |
| 手機切換到 IPv6 地址後無法連接 | 服務端 IPv6 防火牆未放行端口 | 登入境外路由器，在 IPv6 防火牆中新增入站規則，放行 TCP 端口 8443（注意 IPv6 無需 NAT）。 |
| 境外服務端沒有公網 IPv6 地址 | 寬頻運營商未分配 IPv6 | 改用 IPv4 連接，或向運營商申請開通 IPv6 雙棧。 |
| 內地客戶端無法解析 IPv6 地址 | DNS 未返回 AAAA 記錄 | 手動在客戶端填寫 IPv6 地址（而非域名），繞過 DNS 解析。 |


## 6. 客戶端連接與驗證

### 6.1 Windows (v2rayN)

1. 從 GitHub 下載最新版 v2rayN（選擇 `v2rayN-windows-64.zip`），解壓至非系統盤（如 `D:\Tools\v2rayN`）。
2. **導入配置（二選一）**：
   - **方法一：查找分享/導出功能（推薦）**：在 PassWall2 的「節點列表」中，查找每個節點行末或「編輯」界面內是否有 **「分享」**、**「導出」** 或 **「二維碼」** 按鈕。點擊後複製生成的 `vless://` 鏈接，在 v2rayN 中點擊 `伺服器` → `從剪貼板導入` 即可。
   - **方法二：手動輸入（若找不到上述按鈕）**：在 v2rayN 中點擊 `伺服器` → `添加 VMess/VLESS 伺服器`，**手動填入**以下參數（從 PassWall2 節點編輯頁面複製）：
     - **地址 (Address)**：你的域名或 IP
     - **端口 (Port)**：`8443`（與伺服器配置一致）
     - **用戶ID (UUID)**：與伺服器 `config.json` 中的 `id` 一致
     - **流控 (Flow)**：`xtls-rprx-vision`
     - **加密 (Encryption)**：`none`
     - **傳輸協議 (Network)**：`tcp`
     - **偽裝類型 (Type)**：`none`
     - **安全 (Security)**：`reality`
     - **公鑰 (PublicKey)**：伺服器端執行 `xray x25519` 生成的 **公鑰**
     - **ShortId**：與伺服器 `shortIds` 列表中的值一致
     - **指紋 (Fingerprint)**：可填 `chrome` 或 `firefox`
3. 開啟代理：右鍵系統托盤圖標 → 系統代理 → 自動配置系統代理。
4. 測試延遲：右鍵節點 → 測試伺服器真連接延遲，若返回數值則連通。
5. **退出時清理**：在 `設置 (Setting)` → `參數設置 (Parameter Setting)` 中勾選「退出時清除系統代理」，避免關閉後無法上網。

### 6.2 Android (v2rayNG)

1. 下載 v2rayNG（GitHub 或 F-Droid，注意避開 Google Play 上的山寨克隆版）。
2. **導入配置（二選一）**：
   - **方法一：查找分享/導出功能（推薦）**：在 PassWall2 的「節點列表」中，查找「分享」或「二維碼」按鈕。點擊後可使用 v2rayNG 的 **「掃描二維碼」** 功能自動導入。
   - **方法二：手動輸入（若找不到上述按鈕）**：點擊右下角 `+` → 選擇 `手動輸入[VLESS]`，**手動填入**以下參數：
     - **地址 (Address)**：你的域名或 IP
     - **端口 (Port)**：`8443`
     - **用戶ID (UUID)**：與伺服器一致
     - **流控 (Flow)**：`xtls-rprx-vision`
     - **加密 (Encryption)**：`none`
     - **傳輸協議 (Transport)**：`tcp`
     - **偽裝類型 (Type)**：`none`
     - **安全 (Security)**：`reality`
     - **公鑰 (PublicKey)**：伺服器端的 **公鑰**
     - **ShortId**：與伺服器一致
     - **指紋 (Fingerprint)**：可填 `chrome` 或 `firefox`
3. 點擊右下角開關啟動，觀察日誌是否有 `accepted` 字樣。

### 6.3 本節故障排查

> **說明**：以下排查項涵蓋客戶端配置完成後、嘗試連接階段可能遇到的各類問題。

| 問題現象 | 可能原因 | 解決方法 |
| :--- | :--- | :--- |
| 客戶端顯示「已連接」，但瀏覽器無法打開 Google | 系統代理未生效 / 瀏覽器繞過代理 | 檢查 v2rayN 狀態欄是否顯示「系統代理：開啟」；關閉瀏覽器自帶的 DoH（基於 HTTPS 的 DNS）。 |
| 延遲測試返回 `failed` 或超時 | 伺服器端口未放行 / 防火牆阻斷 / 域名解析失敗 | 在內地執行 `telnet [域名] 8443` 確認端口可達；執行 `nslookup [域名]` 確認 DNS 解析正確。 |
| 延遲測試只有 56ms，但實際網頁加載很慢（>60s） | DNS 解析超時 / 路由死循環 | 確保客戶端路由規則為「繞過大陸 (Whitelist)」；檢查 v2rayN 日誌是否有 `dns: failed to lookup` 報錯。 |
| 手機連上了，電腦卻連不上 | 系統時間偏差 / 權限不足 | 檢查電腦系統時間是否偏差超過 5 分鐘；嘗試以管理員身份運行 v2rayN。 |
| v2rayNG 顯示「已啟動」但無法上網 | Android 系統代理設置被其他 App 佔用 | 嘗試切換至「VPN 模式」；或重啟手機。 |
| 電腦關閉 v2rayN 後無法上網 | 系統代理未清除 | 手動進入 Windows 設置 → 網絡和互聯網 → 代理，關閉「使用代理伺服器」開關；或勾選 v2rayN 的「退出時清除系統代理」。 |
| 客戶端日誌出現 `tls: handshake failure` | 系統時間偏差超過 5 分鐘 | 同步系統時間（Windows：設置 → 時間和語言 → 自動設置時間）。 |
| 導入節點後無法點擊「測試延遲」 | 節點配置不完整 / ID (UUID) 格式錯誤 | 重新從伺服器端導出 `vless://` 鏈接，或手動檢查所有字段是否填寫完整。 |
| v2rayN 左下角顯示「Xray 核心未運行」 | 權限不足 / 路徑包含中文 | 以管理員身份運行；確保解壓路徑不含中文字符。 |

**在哪看日誌？**  
- **v2rayN**：主界面 → 選中節點 → 點擊下方「日誌」標籤頁。  
- **v2rayNG**：主界面 → 點擊左上角三條橫線 → 點擊「日誌」。  
複製最後幾行報錯信息進行搜索，是解決此類問題的最快方式。

### 6.4 快速健康檢查（三步驗證）

配置完成後，請務必執行以下三步檢查，以確保方案生效：

**第一步：確認鏈路已通**
- **客戶端看**：v2rayN 節點延遲測試返回具體數字（如 `56ms`），而非 `-1` 或 `超時`。
- **服務端看**：PassWall2 節點列表該節點前有**綠色勾號 (√)** 或綠點，表示節點狀態正常。

**第二步：確認代理已生效**
- 瀏覽器訪問 `http://ip.sb`，頁面顯示的 IP 地址**必須是你境外服務器的公網 IP**。如果顯示的是你內地的寬頻 IP，說明分流規則有誤，流量並未經過代理。

**第三步：確認分流策略正確**
- 訪問一個純內地網站（如 `http://ipip.net`），應能瞬間打開，且顯示的是你內地的 IP。這證明內地流量正在「繞過」代理直連。

若以上三步中任何一步未通過，請返回對應的「故障排查」章節進行診斷。


## 7. 附錄：中國內地網絡環境下的軟件源配置

### 7.1 為什麼需要更換軟件源？

在中國內地網絡環境中，直接訪問 OpenWrt 官方軟件源（`downloads.openwrt.org`）以及 GitHub 等代碼託管平台，通常會遇到**連接超時、下載速度極慢或無法訪問**的情況。這是因為這些服務器的域名或 IP 地址在國內受到網絡訪問限制。

為了解決這一問題，國內多所高校和企業運營了**開源軟件鏡像站**，這些鏡像站定期與官方源同步，提供完整的 OpenWrt 軟件包下載服務。將 OpenWrt 的包管理器（`opkg`）指向這些國內鏡像站，可以**顯著提升下載速度和成功率**。

### 7.2 推薦的國內開源鏡像站

以下三個鏡像站是國內最主流、最穩定的 OpenWrt 軟件源選擇：

| 鏡像站名稱 | 鏡像站地址 | 特點 |
| :--- | :--- | :--- |
| **清華大學開源軟件鏡像站 (TUNA)** | `https://mirrors.tuna.tsinghua.edu.cn/openwrt/` | 速度快、穩定性高，由清華大學學生網絡管理協會運營。 |
| **中國科學技術大學開源鏡像站 (USTC)** | `http://mirrors.ustc.edu.cn/openwrt/` | 教育網訪問極快，由中科大 Linux 用戶協會運營。 |
| **阿里雲開源鏡像站** | `https://mirrors.aliyun.com/openwrt/` | 商業雲服務商運營，帶寬充足，穩定性好。 |

> **選擇建議**：如果你身處教育網環境（高校校園網），優先選擇**清華大學 TUNA** 或**中科大 USTC** 鏡像站；如果在家庭寬頻或商業網絡環境，**阿里雲**鏡像站通常有更穩定的表現。

### 7.3 操作步驟

以下操作需要通過 SSH（安全外殼協議）登錄到 OpenWrt 路由器後執行。

**第一步：備份原始配置文件**

```bash
cp /etc/opkg/distfeeds.conf /etc/opkg/distfeeds.conf.bak
```

**第二步：修改軟件源地址**

使用 `sed` 命令將官方源地址批量替換為鏡像站地址。以下以替換為**清華大學 TUNA 鏡像站**為例：

```bash
sed -i 's/downloads.openwrt.org/mirrors.tuna.tsinghua.edu.cn\/openwrt/g' /etc/opkg/distfeeds.conf
```

如需使用其他鏡像站，將上述命令中的 `mirrors.tuna.tsinghua.edu.cn/openwrt` 替換為對應地址即可。

**第三步：更新軟件包列表**

```bash
opkg update
```

若執行後無報錯，即表示軟件源已成功切換至國內鏡像站。

**第四步：恢復官方源（如需）**

```bash
cp /etc/opkg/distfeeds.conf.bak /etc/opkg/distfeeds.conf
opkg update
```
